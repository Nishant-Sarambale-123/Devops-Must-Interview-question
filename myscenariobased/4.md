## ✅ **Scenario 4: Terraform Infrastructure Drift (Production)**

**Answer:**

### **1️⃣ How I Would Identify and Confirm Infrastructure Drift**

First, since this is **production**, I would avoid making any immediate changes.

I would:

*   Run `terraform plan` in **read‑only mode**
*   Carefully review:
    *   Which resources are changing
    *   Whether the changes are destructive or additive
*   Compare Terraform state with actual AWS infrastructure using:
    *   AWS Console
    *   AWS CLI
*   Check the Terraform state file:
    *   `terraform state list`
    *   `terraform state show <resource>`

✅ My goal is to **confirm whether the drift is real**, understand its scope, and ensure Terraform is not about to unintentionally destroy production resources.

***

### **2️⃣ Common Reasons for Terraform State Drift**

From experience, the most common reasons are:

*   ✅ **Manual changes made directly in AWS**
    *   Through Console or CLI
*   ✅ **Resources created outside Terraform**
    *   By another pipeline or team
*   ✅ **Incorrect or missing state file updates**
*   ✅ **IAM permission changes**
    *   Terraform can’t read actual resource config
*   ✅ **Race conditions**
    *   Multiple engineers or pipelines running Terraform simultaneously

This usually points to **lack of governance or access control** rather than Terraform itself.

***

### **3️⃣ How I Would Fix This Safely in Production**

My approach is always **safety first**:

*   I would **not apply changes blindly**
*   If Terraform is missing existing resources:
    *   Use `terraform import` to bring them into state
*   If attributes were changed manually:
    *   Update Terraform code to reflect actual desired state
*   For critical resources:
    *   Validate changes in **lower environments first**
*   If the changes are risky:
    *   Coordinate with stakeholders
    *   Schedule a controlled maintenance window

✅ The key principle: **Terraform state must represent reality before applying changes**.

***

### **4️⃣ Preventive Measures & Best Practices**

To prevent future drift, I would implement:

✅ **Strict access controls**

*   Restrict AWS Console access
*   Terraform‑only changes via CI/CD

✅ **Remote state with locking**

*   Use S3 + DynamoDB for state locking

✅ **Single source of truth**

*   No manual infra changes allowed

✅ **Pipeline‑based Terraform execution**

*   No local applies to production

✅ **Regular drift detection**

*   Scheduled `terraform plan` in read‑only mode

✅ **Clear ownership and change management**

*   PR reviews for Terraform code
*   Change approvals for production infra
