## ✅ **Scenario: Zero‑Downtime Kubernetes Deployment**

**Answer:**

### **1️⃣ Deployment Strategy I Would Choose & Why**

For a **business‑critical application with zero downtime requirement**, I would choose a **Canary Deployment strategy**.

**Why Canary:**

*   Allows releasing the new version to a **small percentage of users**
*   Reduces blast radius in case of issues
*   Enables **real‑time validation** of the new version under production traffic
*   Safer than rolling updates and more cost‑effective than full blue‑green in large systems

For extremely high‑risk releases, I may combine **Canary + Fast Rollback**, or use **Blue‑Green** for one‑click revert—but Canary is my default choice.

***

### **2️⃣ How I Would Implement It Step by Step**

**Step 1: CI/CD Preparation**

*   Jenkins / GitHub Actions builds a **new Docker image**
*   Image tagged with:
    *   Git commit SHA or version
*   Helm values updated for canary release

***

**Step 2: Deploy Canary via Helm**

*   Deploy a new Helm release or update existing chart:
    *   Canary version starts with **5–10% traffic**
*   In Kubernetes:
    *   Create a separate Deployment (e.g., `app-canary`)
    *   Ensure proper labels and selectors

***

**Step 3: Traffic Steering via ALB**

*   Use ALB target group weighting or ingress annotations
*   Route limited traffic to canary pods
*   Keep stable version serving majority of traffic

✅ User experience remains unaffected for most users.

***

### **3️⃣ Monitoring & Decision Making**

I would closely monitor:

✅ **Application Metrics**

*   Error rate (4xx/5xx)
*   Latency (P95/P99)
*   Throughput

✅ **Infrastructure Metrics**

*   CPU / memory
*   Pod restarts
*   HPA behavior

✅ **Business Metrics**

*   Login success
*   Transactions / conversions (if applicable)

**Decision logic:**

*   If metrics stay healthy → gradually increase traffic (10% → 50% → 100%)
*   If degradation is detected → immediate rollback

Rollback is fast because:

*   Old version is already running
*   Just redirect traffic back

***

### **4️⃣ Rollback Strategy**

Rollback options:

*   Helm rollback to previous stable revision
*   Shift ALB traffic back to stable deployment
*   No pod recreation required

✅ This ensures rollback within minutes.

***

### **5️⃣ What Could Go Wrong & How I Handle It**

| Risk                   | Mitigation                             |
| ---------------------- | -------------------------------------- |
| Hidden performance bug | Canary limits impact                   |
| Config mismatch        | Config validated in lower environments |
| Slow detection         | Alerts + dashboards                    |
| Human error            | Automated pipelines + approvals        |

***

### ✅ **Strong Interview Closing Statement**

> “For zero‑downtime production deployments, I prefer Canary releases because they minimize risk, provide real‑time confidence, and allow instant rollback—ensuring both reliability and business continuity.”
