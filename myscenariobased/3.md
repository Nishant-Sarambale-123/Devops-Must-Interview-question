## ✅ **Scenario 3: Kubernetes Scaling & Performance Issue**

**Answer:**

### **1️⃣ How I Would Troubleshoot This Step by Step**

Since the application is **slow but not down**, I would treat this as a **performance degradation incident**.

**Step 1: Validate the user impact and scope**

*   Confirm which APIs or services are slow
*   Check ALB metrics:
    *   `TargetResponseTime`
    *   `RequestCountPerTarget`
*   Correlate peak traffic times with latency spikes

✅ Purpose: Understand whether slowness is traffic‑driven or internal.

***

**Step 2: Analyze Kubernetes pod health**

*   Check pod status using:
    *   `kubectl get pods`
*   Review pod restarts and readiness
*   Ensure traffic is evenly distributed across pods

✅ Purpose: Running pods does not always mean healthy application behavior.

***

**Step 3: Check HPA configuration and metrics**

*   Review HPA status:
    *   `kubectl describe hpa`
*   Confirm:
    *   Current replicas vs max replicas
    *   Target CPU thresholds
    *   Metrics availability

✅ Purpose: Identify why scaling is not triggering.

***

**Step 4: Analyze resource usage**

*   Review Grafana dashboards:
    *   CPU / Memory utilization
    *   Request vs limit mismatches
*   Check node capacity and scheduling status

✅ Purpose: CPU saturation or memory pressure can slow response times.

***

### **2️⃣ Why HPA Might Not Be Scaling Despite High CPU**

From experience, the most common reasons are:

*   **Missing or incorrect CPU requests**
    *   HPA uses CPU *requests*, not limits
*   **Metrics server issues**
    *   HPA cannot scale if metrics are unavailable
*   **Max replica limit reached**
    *   HPA cannot scale beyond configured maximum
*   **CPU spikes but averaging hides the signal**
    *   Short‑term spikes don’t trigger scaling
*   **Application is memory‑bound instead of CPU‑bound**

***

### **3️⃣ Immediate Actions to Reduce User Impact**

My priority would be to **stabilize performance quickly**:

*   Manually scale replicas temporarily
*   Increase max replicas in HPA if needed
*   Add more worker nodes if cluster is saturated
*   Throttle or rate‑limit non‑critical traffic
*   Communicate temporary mitigation to stakeholders

✅ Goal: Improve response time while deeper analysis continues.

***

### **4️⃣ Long‑Term Improvements and Preventive Measures**

To avoid recurrence, I would recommend:

*   ✅ Proper CPU and memory requests & limits
*   ✅ Custom HPA metrics (latency / RPS‑based scaling)
*   ✅ Autoscaling using KEDA if event‑driven
*   ✅ Load testing to validate scaling behavior
*   ✅ Better observability and alerting on saturation
*   ✅ Capacity planning for peak hours
